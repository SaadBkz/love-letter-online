- **Tests existants** : `tests/game/*.test.ts` — 41 tests verts (baseline).
- **Simulation engine** : `tests/game/simulation.test.ts` (créé pour cet audit) — **250 parties complètes** à 4 bots, 14194 actions, 745 manches, **0 crash**, **0 violation d'invariant**. Invariants vérifiés à chaque step :
  - I1 : conservation `sum(hands)+sum(discards)+deck+removed+publicRemoved == 21`.
  - I2 : `currentPlayerIdx` pointe un non-éliminé pendant `turnPhase='play'`.
  - I3 : `turnPhase='resolvingChancellor' ⇔ chancellorHand !== null`.

## Cartographie du flux multijoueur

```
[Client A] tap carte
    ↓
submitAction() → POST /api/rooms/[code]/actions
    ↓
updateRoom(code, room => applyAction(room.state, action))
    ├── relit Redis           ← optimistic, no CAS
    ├── applyAction (pure)
    └── setRoom(updated)
         ↓
    pusher.trigger(`room-updated-${pid}`) × N joueurs
         ↓
    [Clients A..N] reçoivent room-updated-{leur-id}
         ↓
    if (data.version > versionRef.current) setRoom(...)
```

Points sensibles repérés :

- **fetchRoom au mount** (page.tsx:48) et **events Pusher** (page.tsx:84-91) coexistent → race possible.
- **`updateRoom`** (rooms.ts:43) n'est PAS atomique malgré le commentaire (`// pour une vraie atomicité, il faudrait CAS via Lua`).
- **Aucun re-fetch après reconnexion Pusher** : si le client perd la connexion Pusher pendant 30s, il ne récupère pas le state manqué.

## Bugs identifiés

### B1 — Race sur `room-updated` après refresh : version d'état perdue ou prise dans le mauvais ordre

- **Sévérité** : critical
- **Catégorie** : multiplayer / refresh
- **Reproduction** :
  1. Joueur humain refresh (F5) pendant le tour d'un bot/autre joueur.
  2. `useEffect` au mount lance `fetchRoom` (page.tsx:48) ET `subscribeRoom` (page.tsx:65) en parallèle.
  3. Si Pusher se connecte rapidement et reçoit `room-updated-X` AVANT que le `fetchRoom` initial ne résolve, alors :
     - L'event Pusher passe par le handler `onRoomUpdated`, met `versionRef.current = data.version` et set le state.
     - Quand le `fetchRoom` initial résout (page.tsx:50-53), il fait `setRoom(r); versionRef.current = r.version` SANS comparer — il **écrase** potentiellement avec un state plus ancien.
- **Évidence** :
  ```ts
  // app/play/[code]/page.tsx:48-54
  fetchRoom(identity)
    .then((r) => {
      if (!alive) return;
      setRoom(r);
      versionRef.current = r.version;  // ← écrase sans comparer !
    })
  ```
  vs.
  ```ts
  // ligne 84-91
  const onRoomUpdated = (data: { state: GameState; version: number }) => {
    if (data.version > versionRef.current) {  // ← protégé par >
      versionRef.current = data.version;
      setRoom(...);
    }
  };
  ```
- **Cause racine** : asymétrie. Le fetch initial accepte tout, l'event Pusher ne prend que ce qui est plus récent. Si l'event arrive en premier, on écrase avec du périmé.
- **Correction** : dans le `.then` du fetchRoom, comparer la version : `if (r.version > versionRef.current) { setRoom(r); versionRef.current = r.version; }`.

### B2 — `currentPlayerIdx` peut pointer un éliminé en `lastRoundSummary` (refresh + Princesse)

- **Sévérité** : high
- **Catégorie** : engine + UI
- **Reproduction** :
  1. Le joueur courant joue Princesse → s'auto-élimine → `applyAction` appelle `checkEndOfRound` → `endRound` retourne un state dont `currentPlayerIdx` pointe TOUJOURS sur le joueur qui vient de s'éliminer (Princesse) ou sur la dernière action.
  2. `endRound` (end-of-round.ts:96-105) ne réinitialise PAS `currentPlayerIdx`.
  3. Si un client refresh à ce moment précis (avant que `startNextRound` ne soit appelé), il reçoit un state où `state.players[state.currentPlayerIdx].isEliminated === true`.
  4. `GameTable.tsx:59` fait `const current = state.players[state.currentPlayerIdx]!;` puis `isHumanTurn = current.id === humanId && state.turnPhase === 'play'`. `state.turnPhase` est `'play'` après `endRound`. Si `humanId === current.id` et `current.isEliminated`, alors `isHumanTurn === true` mais le joueur ne peut rien faire (sa main est vide).
- **Évidence** :
  ```ts
  // lib/game/end-of-round.ts:96-105 — NE TOUCHE PAS currentPlayerIdx
  return {
    ...state,
    players,
    turnPhase: 'play',
    chancellorHand: null,
    lastRoundSummary: summary,
    ...
  };
  ```
  ```ts
  // components/game/GameTable.tsx:59-60
  const current = state.players[state.currentPlayerIdx]!;
  const isHumanTurn = current.id === humanId && state.turnPhase === 'play';
  ```
  Le test `tests/game/end-of-round.test.ts:189-210` confirme que ce cas existe et qu'il faut autoriser `startNextRound` même quand le current est éliminé.
- **Cause racine** : `endRound` laisse `currentPlayerIdx` tel quel ; `RoundEndModal` est censé masquer le `GameTable`, mais comme le modal est rendu PAR-DESSUS et non bloquant strictement, le state du table reste ambigu. De plus, `isHumanTurn` peut être faussement vrai si le humain s'est suicidé Princesse.
- **Correction** : dans `endRound`, soit avancer `currentPlayerIdx` au prochain non-éliminé, soit l'ignorer en UI tant que `lastRoundSummary !== null`. La logique `isHumanTurn` dans GameTable devrait inclure `&& !state.lastRoundSummary && !current.isEliminated`.

### B3 — Aucune indication visuelle dédiée "C'est mon tour" dans le seat humain

- **Sévérité** : critical (bug signalé par l'utilisateur : "les joueurs ne savent pas au tour de qui c'est de jouer")
- **Catégorie** : UI
- **Reproduction** :
  1. Lancer une partie en multi à 4. Observer la zone basse (LocalHand) et la zone haute (PlayerSeat des adversaires).
  2. Quand c'est mon tour, le seat humain n'existe PAS — la barre du bas s'éclaire (`border-[color:var(--color-gold-bright)]` + `boxShadow`), mais c'est subtil et le pseudo n'est mentionné que dans le tout petit header (line GameTable.tsx:243).
  3. Quand c'est le tour d'un AUTRE, le `PlayerSeat` de cet adversaire glow (`isCurrent` boxShadow animé). Mais aucun texte clair "C'est le tour de X" n'apparaît.
- **Évidence** :
  ```ts
  // components/game/GameTable.tsx:236-307 — le humain n'a pas de seat
  // Ligne 271-284 : seuls les `opponents` ont un PlayerSeat avec glow.
  // Ligne 295-307 : la zone du humain a juste une bordure dorée discrète.
  ```
- **Cause racine** : le design "humain = main en bas / adversaires = bandeaux du haut" dilue l'indicateur de tour. Le glow du PlayerSeat est limité aux adversaires. Le humain ne voit pas un badge clair "À toi !" et ne sait pas qui joue actuellement (juste un glow sur l'avatar de l'adversaire courant, parfois confondu avec la mise en valeur générale).
- **Correction** : ajouter une bannière compacte sous le header (genre `<div>Tour de {current.name}</div>`) qui change de couleur quand `current.id === humanId` (vert/or pulsé), avec animation ; ou un fil d'Ariane visuel "› À toi ›".

### B4 — `TargetPickerModal` / `GuardGuessModal` peuvent rester ouverts si la manche se termine ou que le tour change pendant l'interaction

- **Sévérité** : high
- **Catégorie** : UI / multiplayer
- **Reproduction** :
  1. Humain clique Garde, le `TargetPickerModal` s'ouvre (`pendingCard='Guard'`).
  2. Avant qu'il choisisse une cible, un autre événement arrive (par ex. le humain se fait éliminer par un Prince via Pusher event ; ou quelqu'un joue, ce qui ne devrait pas arriver hors tour, mais peut arriver si l'humain rafraîchit pendant un tour de bot).
  3. Le `pendingCard` n'est PAS reset par effet ; le modal reste affiché. Si le humain pousse `pendingCard='Guard'`, target='X', guess='Priest', le `submitAction` retourne erreur "Action refusée" puisque ce n'est plus son tour.
- **Évidence** :
  ```ts
  // components/game/GameTable.tsx:64-66
  const [pendingCard, setPendingCard] = useState<CardKind | null>(null);
  const [pendingTarget, setPendingTarget] = useState<PlayerId | null>(null);
  ```
  Aucun `useEffect` ne fait `setPendingCard(null)` quand `state.currentPlayerIdx` change ou `state.lastRoundSummary !== null`.
- **Cause racine** : pas de désynchro entre state local UI et state du jeu.
- **Correction** : ajouter `useEffect(() => { if (!isHumanTurn) { setPendingCard(null); setPendingTarget(null); } }, [isHumanTurn]);`.

### B5 — `processedLogCountRef` initialisé sur `state.log.length` rate les reveals si le state arrive après le mount

- **Sévérité** : medium
- **Catégorie** : multiplayer / UI
- **Reproduction** :
  1. À chaque mount du composant `GameTable`, `processedLogCountRef = useRef(state.log.length)` (line 102) capture la longueur initiale.
  2. Or, en multijoueur, le PREMIER state reçu peut déjà contenir des reveals (refresh en milieu de manche). Aucune bulle ne sera rejouée — c'est l'effet voulu : pas de spam au refresh.
  3. **Mais** : à chaque navigation interne (par ex. close/reopen `LogDrawer`, déclenché par un re-render parent), un nouveau mount peut survenir. L'utilisateur peut "manquer" un reveal s'il bouge dans l'arborescence à un mauvais moment.
  4. Plus subtil : quand le composant se monte avec `state.log.length === 0` (état avant que le state arrive vraiment, via un refresh très rapide), le ref est figé à 0. Quand le state arrive, le ref est encore 0, donc tous les reveals d'historique seront rejoués → spam de bulles.
- **Évidence** :
  ```ts
  // components/game/GameTable.tsx:102
  const processedLogCountRef = useRef(state.log.length);
  ```
  Il n'y a aucune mise à jour de ce ref tant qu'aucun nouvel event n'arrive.
- **Cause racine** : `useRef` ne se re-init pas entre rendus, mais sa valeur initiale dépend de la prop au TOUT premier rendu. Si la prop est artificiellement vide, on rate la mise au point.
- **Correction** : initialiser à 0 et faire un effet dédié au tout premier render pour set à `state.log.length` quand on est sûr que le state est complet (par ex. par identifiant de partie).

### B6 — `RoundEndModal` peut s'afficher pour des clients qui n'ont pas reçu le state final si Pusher trigger échoue partiellement

- **Sévérité** : critical (bug signalé : "fin de manche bug avant prochaine manche")
- **Catégorie** : multiplayer
- **Reproduction** :
  1. Un joueur joue la dernière action de la manche → `applyAction` retourne un state avec `lastRoundSummary !== null`.
  2. `app/api/rooms/[code]/actions/route.ts:56-64` itère sur `updated.players` et fait `await getPusher().trigger(...)` pour chacun, dans une **boucle séquentielle**.
  3. Si la 1ʳᵉ ou 2ᵉ trigger réussit mais la 3ᵉ ou 4ᵉ échoue (rate limit Pusher, timeout réseau), les clients qui n'ont pas reçu l'event continuent à voir l'ancien state — **pas** de RoundEndModal pour eux.
  4. Quand un client cliquant "Manche suivante" déclenche une nouvelle action, le serveur retourne erreur 422 "Action invalide: Pas de manche à démarrer" car son state local est désynchronisé.
- **Évidence** :
  ```ts
  // app/api/rooms/[code]/actions/route.ts:56-64
  if (isPusherConfigured() && updated.state) {
    for (const p of updated.players) {
      const view = filterStateForPlayer(updated.state, p.id);
      await getPusher().trigger(CHANNEL(code), `${EVENTS.roomUpdated}-${p.id}`, {
        state: view,
        version: updated.version,
      });
    }
  }
  ```
  Aucun catch — si une trigger throw, la suite du loop est court-circuitée et la réponse client retourne 500 (mais le state Redis a déjà été mis à jour).
- **Cause racine** : 1 commit Redis + N triggers Pusher SANS atomicité, sans retry, sans fallback poll-on-stale.
- **Correction** : (a) wrapper chaque trigger dans try/catch indépendant ; (b) déclencher un seul event "room-version" générique et laisser chaque client GET son state via `fetchRoom` ; (c) ajouter un poll de fallback côté client toutes les 10s si pas d'event reçu.

### B7 — Race condition `startNextRound` : deux clients cliquent simultanément, un succès + un état refusé silencieux ou rollback

- **Sévérité** : high
- **Catégorie** : multiplayer
- **Reproduction** :
  1. Tous les joueurs voient le `RoundEndModal` ; tous ont un bouton "Manche suivante" actif.
  2. Joueurs A et B cliquent quasi simultanément.
  3. Premier appel API : `applyAction(state, {kind:'startNextRound'})` → state avance en R+1, `lastRoundSummary` redevient null.
  4. Deuxième appel API : refait `applyAction` sur le state Redis qui est ENTRE les deux états (lecture optimiste). Si le 1er n'a pas encore commit son `setRoom`, le 2e va appliquer `startNextRound` une 2ᵉ fois → la manche est relancée et la 1ʳᵉ est écrasée.
  5. Ou bien le 2e voit déjà `lastRoundSummary === null` (si commit du 1er passé) et la `validateAction` retourne "Pas de manche à démarrer" → toast d'erreur sur le client B alors qu'il pensait juste passer.
- **Évidence** :
  ```ts
  // lib/store/rooms.ts:48-58
  for (let i = 0; i < maxRetries; i++) {
    const current = await getRoom(code);
    if (!current) return null;
    const next = fn(current);
    if (!next) return current;  // ← retourne current sans incrément version, mais le client pensait avoir update
    const updated: Room = { ...next, version: current.version + 1 };
    await setRoom(updated);
    return updated;
  }
  ```
  Pas de CAS, pas de WATCH/MULTI/EXEC. Une écriture concurrente perd ou cause un état surprenant.
- **Cause racine** : optimistic-lock fait sans verrou réel, juste un retry sans détection de conflit.
- **Correction** : utiliser Lua script Upstash + version check, OU restreindre `startNextRound` au gagnant de la manche/à l'hôte uniquement (UI désactive le bouton pour les autres avec un texte "{gagnant} démarre la prochaine manche").

### B8 — `updateRoom` n'est pas atomique : `maxRetries=3` mais aucune CAS — perte d'écriture possible sous contention

- **Sévérité** : high
- **Catégorie** : multiplayer
- **Reproduction** : voir B7. Le code est commenté lui-même : `// Note : pour une vraie atomicité, il faudrait CAS via Lua. Ici on s'appuie sur le fait que les actions de jeu sont routées par joueur courant (faible contention par salle).` Mais `startNextRound` est explicitement DÉ-routé du joueur courant (`validateAction` line validation.ts:15-20), donc plusieurs joueurs peuvent envoyer en parallèle.
- **Évidence** : `lib/store/rooms.ts:43-60`. Le `for` boucle avec `maxRetries=3` mais ne re-vérifie jamais que le state lu n'a pas changé. La 2ᵉ écriture concurrente écrase la 1ʳᵉ.
- **Cause racine** : pas d'atomicité applicative.
- **Correction** : Upstash supporte `eval` (Lua). Implémenter un script `room:update` qui fait GET → check version → SET dans un seul aller-retour.

### B9 — `RoundEndModal` peut bloquer le humain si lui-même est éliminé, puisque le state ne lui montre que sa main vide

- **Sévérité** : medium
- **Catégorie** : UI
- **Reproduction** :
  1. Humain joue Princesse (s'élimine). `RoundEndModal` s'affiche.
  2. Le humain voit son `finalHands[humanId] === null` (car éliminé : end-of-round.ts:27 `finalHands[p.id] = p.isEliminated ? null : (p.hand[0] ?? null);`).
  3. Le modal affiche "éliminé·e" pour lui (RoundEndModal.tsx:75-82), ce qui est correct visuellement mais perturbant UX : aucune réelle indication de "tu as perdu cette manche, voici qui a gagné" — juste le badge ✓ sur le seat du gagnant.
  4. Si c'est `gamePhase === 'ended'` (le gagnant atteint TOKENS_TO_WIN), `showRoundEnd` est `false` (page.tsx:181-183) car `winnerId !== null` filtré explicitement, donc seul le `GameEndModal` s'affiche : OK pour la fin de partie, mais entre les deux états il y a un flash possible.
- **Évidence** :
  ```ts
  // app/play/[code]/page.tsx:180-184
  const showRoundEnd =
    room.state.lastRoundSummary !== null &&
    room.state.gamePhase === 'playing' &&
    room.state.winnerId === null;
  const showGameEnd = room.state.gamePhase === 'ended';
  ```
  Si l'event Pusher arrive avec un state final (winnerId !== null), `showRoundEnd` devient false et `showGameEnd` true au même rendu : OK. Mais si une régression introduit un state intermédiaire (winnerId set mais gamePhase encore "playing"), les deux modals peuvent se chevaucher.
- **Cause racine** : checks dérivés sur `state` au lieu d'un seul flag explicite "endOfRoundAck".
- **Correction** : exposer un enum dérivé strict, ou consolider en `<EndScreens state={state} />` qui contient la logique en un endroit.

### B10 — `versionRef.current === r.version` : le `>` strict refuse de remplacer un state stale par un state au même numéro après reconnexion Pusher

- **Sévérité** : medium
- **Catégorie** : refresh / multiplayer
- **Reproduction** :
  1. Joueur déconnecte sa wifi (Pusher disconnect).
  2. Un autre joueur joue → version Redis incrémentée à V+1, mais l'event Pusher est perdu.
  3. Joueur revient en ligne : Pusher reconnecte automatiquement, mais aucun mécanisme ne re-fetch le state. `versionRef.current` est resté à V.
  4. Le prochain event reçu a `data.version === V+2` (par exemple) → OK, le check `>` accepte.
  5. **Mais** si l'utilisateur F5 entretemps et que le fetch retourne `V+1` (cohérent avec ce qu'il s'est passé en son absence), tout va bien. Si l'utilisateur ne refresh pas et que NOTRE version est restée à V, tous les events SUBSÉQUENTS arrivent comme `V+1, V+2, V+3` etc., on les voit, mais la transition V → V+1 fait un saut visuel : on rate les reveals intermédiaires.
- **Évidence** :
  ```ts
  // app/play/[code]/page.tsx:84-91
  const onRoomUpdated = (data: { state: GameState; version: number }) => {
    if (data.version > versionRef.current) {
      versionRef.current = data.version;
      setRoom((prev) => prev ? {...prev, state: data.state, version: data.version, gameStarted: true} : prev);
    }
  };
  ```
  Aucun handler pour `pusher:state_change` ou `pusher:subscription_succeeded` pour re-fetch.
- **Cause racine** : pas de réconciliation après reconnexion.
- **Correction** : binder `client.connection.bind('connected', () => fetchRoom(...))` et re-pull le state à chaque reconnexion.

### B11 — `chancellorHand` filtré pour les non-actifs mais `turnPhase==='resolvingChancellor'` reste : aucun client ne sait que tout le monde est bloqué

- **Sévérité** : medium
- **Catégorie** : UI / multiplayer
- **Reproduction** :
  1. Joueur A (autre que humain) joue Chancelière → state passe en `turnPhase='resolvingChancellor'`.
  2. Server broadcast à tous, `filterStateForPlayer` (views.ts:24-29) cache `chancellorHand` aux non-A.
  3. Côté humain : `state.turnPhase === 'resolvingChancellor'` mais `current.id !== humanId` → `isHumanChancellor === false`, le ChancellorModal n'apparaît pas. C'est correct.
  4. **Mais** rien dans `GameTable` n'affiche "A choisit ses cartes (Chancelière)…". Le humain attend sans contexte. Le `BotThinking` (GameLogFeed.tsx:107-110) ne s'affiche que si `isBotTurn === true` (GameTable.tsx:62, qui exige `state.turnPhase === 'play'`). Donc en `resolvingChancellor`, aucun "thinking dot" — le humain croit que la partie est freezée.
- **Évidence** :
  ```ts
  // components/game/GameTable.tsx:62
  const isBotTurn = !!current?.isBot && state.turnPhase === 'play' && !current.isEliminated;
  ```
  ```ts
  // components/game/GameTable.tsx:288-291
  <GameLogFeed log={state.log} thinkingBotName={isBotTurn ? current.name : null} />
  ```
- **Cause racine** : `isBotTurn` exclut explicitement `resolvingChancellor`.
- **Correction** : étendre à `isBotTurn = !!current?.isBot && (state.turnPhase === 'play' || state.turnPhase === 'resolvingChancellor') && !current.isEliminated`. Et adapter le label : "X choisit (Chancelière)…".

### B12 — `revealQueue` peut perdre des reveals si plusieurs sont ajoutés dans le même tick et un setTimeout déjà actif n'est pas annulé

- **Sévérité** : medium
- **Catégorie** : UI
- **Reproduction** :
  1. Une action déclenche plusieurs reveals dans la même `applyAction` (ex : Baron qui révèle + élimine, puis fin de manche qui ajoute d'autres logs).
  2. `useEffect` (GameTable.tsx:104-121) push tous dans la queue (correct).
  3. Mais le `useEffect` d'auto-dismiss (lines 126-139) dépend de `[revealQueue]` et démarre un `setTimeout`. Si un nouveau reveal arrive AVANT que le setTimeout ne s'écoule, l'effet RE-démarre (cleanup + re-fire), donc le 1er reveal est dismiss prématurément ou immédiatement.
  4. Cas concret : Baron compare → reveal "baronCompare" (3300ms). Pendant ce temps, le tour avance, le bot suivant joue Garde → reveal "guardGuess" (3000ms). Le `revealQueue` était `[baron]` → devient `[baron, guard]`. L'effet re-fire avec `head=baron`, durée 3300, mais on a déjà attendu 1.5s. On wait 3300 NOUVELLES ms → le total est 4800ms pour `baron`, ça décale tout. Pas de drop, mais timing cassé.
  5. Pire scénario : si l'effet dépend de `[revealQueue]` (entier), tout changement de queue trigger un nouveau setTimeout. Si je dismiss baron via clic (`onDismiss={() => setRevealQueue(q => q.slice(1))}`), c'est ok. Mais si j'ajoute un nouveau pendant que le 1er est affiché, je risque un timeout zombie.
- **Évidence** :
  ```ts
  // components/game/GameTable.tsx:126-139
  useEffect(() => {
    if (revealQueue.length === 0) return;
    const head = revealQueue[0]!;
    const duration = head.type === 'guardGuess' ? 3000 : ...;
    const t = setTimeout(() => {
      setRevealQueue((q) => q.slice(1));
    }, duration);
    return () => clearTimeout(t);
  }, [revealQueue]);
  ```
  Le cleanup `clearTimeout(t)` annule bien le timeout précédent, mais le NOUVEAU setTimeout repart à 0 — donc le head reste affiché plus longtemps.
- **Cause racine** : dépendance `[revealQueue]` au lieu de `[revealQueue[0]]` (head).
- **Correction** : changer la dépendance, ou utiliser un ref pour stabiliser.

### B14 — Le humain ne sait pas qu'un autre joueur résout sa Chancelière en multi (pas de "X réfléchit", BotThinking gated sur isBot)

- **Catégorie** : multiplayer / UI

### B15 — Sur reconnexion Pusher, le client n'effectue pas de re-fetch authoritatif : si un event a été manqué, le state diverge

- **Sévérité** : critical (très lié à B6)
- **Catégorie** : multiplayer / refresh
- **Reproduction** :
  1. Coupure réseau brève (5s) sur le client A pendant qu'un autre joueur joue.
  2. Pusher reconnecte automatiquement, mais l'event `room-updated-A` envoyé pendant la coupure est perdu (Pusher Channels n'a pas de buffering par défaut sur les events private).
  3. `versionRef.current` reste à V (ancien). Le client A continue à voir l'ancien state.
  4. Au tour suivant, le client A peut tenter de jouer une carte — `submitAction` → 422 "Action invalide: Ce n'est pas le tour de X" car le tour est déjà passé (le state Redis est en V+2).
  5. Toast d'erreur incompréhensible pour le joueur.
- **Évidence** :
  ```ts
  // lib/realtime/pusher-client.ts:31-56 — aucun handler pour 'state_change' ou 'connected'
  // app/play/[code]/page.tsx:61-103 — useEffect qui binde events mais ne re-fetch pas sur reconnexion
  ```
- **Cause racine** : pas de réconciliation au reconnect.
- **Correction** : voir B10. Bind `pusher:connection_state_change` ET re-fetch quand `connected`.

## Synthèse — top 3 bugs prioritaires

1. **B6 — RoundEndModal/state désynchronisé entre clients après une trigger Pusher partielle** (critical, multiplayer). C'est le bug le plus probablement à l'origine de "fin de manche bug avant prochaine manche". Le humain clique "Manche suivante", le serveur retourne 422 ou le tour ne s'amorce pas. À fixer par : (1) re-fetch de fallback sur erreur 422 ; (2) try/catch indépendant par trigger ; (3) bouton "Manche suivante" désactivé pour tous sauf gagnant.

2. **B3/B13 — Pas d'indicateur clair "C'est ton tour"** (critical, UI). Le bug le plus signalé. La pulsation pioche n'est pas suffisante en multi (4 joueurs) car le humain reste passif entre 3 tours d'attente — il a besoin d'un signal fort quand revient son tour. Solution : badge plein écran "À toi !" pendant 1s + texte permanent "Tour de {current.name}" sous le header, en couleur or si `current.id === humanId`.

3. **B1/B15 — Race au refresh + pas de réconciliation après reconnexion Pusher** (critical, multiplayer/refresh). Cause racine de "après refresh, le tour est inconnu / on rate la fin de manche". Le `fetchRoom` initial n'est pas comparé à la version Pusher, et aucun re-fetch n'a lieu sur reconnexion. Solution combinée : (a) `setRoom` initial gardé seulement si `r.version > versionRef.current` ; (b) `client.connection.bind('connected', refetch)`.

> **Engine** : 250 simulations sans aucun crash ni violation. Les bugs viennent uniquement du **client React + plumbing Pusher/Redis**, pas du moteur de jeu pur. Les WIP en cours (`end-of-round.ts`, `init.ts`, `types.ts`, `validation.ts`) ne sont pas la cause des symptômes signalés — l'engine est solide.
